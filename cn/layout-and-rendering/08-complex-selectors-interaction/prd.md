测试用例 8：复杂选择器与交互状态布局

测试目的：评估 Chrome 对复杂 CSS 选择器的支持及其在用户交互下触发样式变化的表现，确保诸如:has()、:focus-within 等高级选择器正确生效，并观察在 DOM 状态变化时样式重计算的性能影响。此测试验证浏览器对关系选择器和嵌套选择器的兼容性，以及在用户点击、聚焦等操作下布局/样式的即时更新是否正确且高效。

测试特性：采用复杂的 CSS 选择器和伪类，包括：父选择器:has()（Chrome 现已支持）根据子元素状态改变父元素样式，:focus 和:focus-within 用于当元素或其子元素获得焦点时应用样式，以及其他结构性伪类如:nth-child、:not()的组合。利用纯 CSS 实现交互效果，例如通过:has(input:checked)配合隐藏的复选框来切换布局（纯 CSS 手风琴/折叠面板），或使用:focus-within 在表单获得焦点时高亮其容器边框。所有交互均不借助 JavaScript，完全依赖 CSS 状态选择器。

供你参考的测试用例设计：页面包含一个可交互的卡片列表。每个卡片（`<div class="card">`）内部有一个隐藏的复选框`<input type="checkbox" class="toggle">`及其“展开详情”作为触发，以及一个详情面板`<div class="details">`。初始状态下.details 通过 CSS 设置 display: none 隐藏。CSS 利用父选择器实现展开逻辑：.card:has(.toggle:checked) .details { display: block; }，当用户点击使复选框选中时，其所在卡片满足:has 条件，从而内部.details 显示。另定义.card:focus-within { outline: 2px solid #0066cc; }，使当卡片内任一元素获得焦点时，高亮卡片边框。页面还包含一些深层次选择器组合测试，如.list >` li:nth-child(odd):hover >` .title::after，用于验证 Chrome 对复杂选择器语法的解析和应用。

供你参考的测试用例使用方法：加载页面后，首先验证静态样式：复杂选择器应已正确应用（例如列表奇数项的标题有特殊样式等）。然后进行交互操作：点击每个卡片的“展开详情”标签（切换复选框），对应卡片内隐藏的详情内容应立即出现，再次点击应收起，观察.card:has()规则是否每次都即时生效，没有出现状态不同步的情况。接着使用键盘 Tab 在卡片之间切换焦点，确认当焦点进入某卡片时，该卡片容器呈现蓝色边框高亮，移出后高亮消失，符合:focus-within 样式。快速在不同卡片间切换交互，确保布局变化流畅，未发生错乱。性能方面，由于:has()选择器可能在 DOM 变化时增加样式匹配开销，当 DOM 很大且变化频繁时可能影响性能 。在本测试，可打开 DevTools 的 Performance 录制，在快速点击切换详情的过程中查看样式重计算（Recalculate Style）的耗时，评估 Chrome 优化程度。一般少量元素下影响可以忽略；如发现样式计算耗时明显，可提示慎用过于复杂的选择器。在本测试中，总体确认复杂选择器在 Chrome 中工作正确，交互引发的样式和布局变化准确无误，没有因为选择器匹配问题导致样式遗漏或延迟。
