你是一个专业 Web 前端开发者，需要你为一款新浏览器引擎开发一系列 demo，作为测试用例，测试这个浏览器引擎在网页渲染方面的兼容性和性能。

本文档会提供这些测试用例的需求，其中有的会包含可供你参考的设计方案（含使用方法），有的不包含，无论是否包含，你都应该为了满足这个测试用例的需求，重新做专门的设计，然后提供代码实现。

你的设计方案、代码实现，一定要遵循以下全部要求：

1. 每个测试用例都由一或多个独立网页组成，每个网页专注于服务单一的测试需求。
2. 每个网页都只能用原始的 HTML、CSS、JS、Web API 来实现，不要使用任何框架和库（比如 React、Vite、npm 等） 。
3. 只需要兼容最新的 Chromium，不用考虑其他浏览器和旧浏览器版本，不用跨平台。
4. 每个网页的代码和内容都要专注于测试需求，避免其他会带来干扰的代码和内容。
5. 为了方便测试，网页内容中可以包含一些有助于查看和评估测试结果的可视化 UI 设计，以及相关的 JS 代码（所有 UI 功能必须真实可用），但不要因为这些 UI 的实现而干扰测试结果。
6. 代码中的注释都用英文，网页内容中的文字都用中文。
7. 每个页面都由以下元素组成：
   a. 顶栏：测试用例的标题、一句话说明
   b. 工具栏：测试用例在使用过程中需要的各种操作控件、参数控制、状态提示
   c. 测试区域
   d. 解释说明：测试目标，测试方案说明，使用方法，查看/检验/评估结果的方法
8. 每个页面的 UI 设计和视觉风格要统一：
   a. 白色底色
   b. 顶栏撑满页面宽度
   c. 其他区域的保持统一的宽度
9. 每个测试用例都放在一个目录里，目录名是测试用例的编号和简短名称，用 dash 分隔。

## 研发侧⽤例 11 ｜ DRM/受保护视频与 Overlay 呈现路径

> 最⼩ JS 依赖：通过 EME 播放受保护内容（或使⽤受保护内容占位/模拟流）；受环境限制较多，⻚⾯应有降级⽅案。

### 测试目标

验证受保护视频在可⽤平台上是否⾛ overlay/保护链路，评估与普通视频在合成与内存上的差异；观
察进⼊/退出全屏、画中画时的表现。

### 涉及特性

Encrypted Media Extensions（EME）、受保护解码、视频 overlay 平⾯、全屏/画中画。

### 页面结构与要点

- 同⻚显⽰：受保护视频 A、⾮受保护视频 B，分辨率⼀致。
- 提供全屏/退出按钮，切换若⼲次。
- UI 清晰提⽰当前状态（受保护/⾮受保护）。

### 人工观察与建议工具

- DevTools → Media：查看解码与加密状态。
- Performance：观察 overlay 下是否减少合成/拷⻉。
  屏幕：切换时⽆⿊屏/闪烁；窗⼝移动/缩放稳定。

## 屏幕：切换时⽆⿊屏/闪烁；窗⼝移动/缩放稳定。

### 验收标准

- 受保护视频渲染正确、⽆截屏残影；

- 资源占⽤合理（对⽐⾮受保护流）。

- 多实例同时播放（系统许可时）。
